## Learned User Preferences

- When changing this Raycast GitLab extension, preserve existing code style, structure, and behavior unless explicitly requested; do not refactor unrelated patterns or drift from local conventions.
- Run requested terminal, lint, and Raycast dev/debug commands yourself when asked instead of only describing them.
- Communicate in Russian in chat; code comments, UI copy, and docs follow existing project language.
- Never commit or push without explicit user request.
- Async mutations and errors: inline `try`/`catch` with `showToast({ style: Toast.Style.Animated, title })` before the action, `showToast(Toast.Style.Success, …)` after, and `showFailureToast` from `@raycast/utils` in `catch`—no shared toast wrapper helpers. Skip for Refresh/navigation-only actions.
- Preferences: use typed `getPreferences()` from `utils.ts` (`Preferences` interface with all `package.json` keys); no per-key helpers or casts at call sites.
- API/GraphQL types and selections: include only fields used in UI and actions; do not expand response schemas with unused properties during refactors.
- Data fetching: prefer `useCachedPromise`, `useCachedState`, and `Cache` from `@raycast/utils` over custom cache hooks; prefer `usePromise` over `useState`+`useEffect` when async fetch is the only state update path; pass `initialData` to `useCachedPromise` where possible (e.g. `[]` for lists) to simplify UI and avoid nullable data guards; handle errors inline in UI (not toast-only `useEffect`)—`usePromise`/`useCachedPromise` show default error toasts unless errors are handled at the call site without suppressing them; do not add custom `onError` on these hooks solely to show toasts. Do not poll list views with `useInterval` or similar timers—refresh via explicit actions only. Menu Bar commands keep Raycast's built-in command `interval` in `package.json` (`mrmenu`, `issuemenu`, `todomenubar`).
- React style: declare JSX inline—do not store elements in `useMemo`; PascalCase for functions that return components; inline 1–3 line single-use expressions (except enum/mapping tables); omit redundant guards when types/API contracts already guarantee values. Do not extract one-off 1–3 line logic into named helpers (`flatListViewPreferences`, `getIcon`, `clipText`, `navTitle`, `stateColor`, …)—put it at the call site instead. Keep a helper only when reused, a mapping table, or the logic is long enough that a name genuinely helps.
- Locals: do not introduce intermediate variables for obvious single-use values—inline them at the call site (e.g. pass `` `mr_pipelines_${mr.project_id}_${mr.iid}` `` directly as `cacheKey` instead of `const cacheKey = ...`). Prefer a single property chain (`foo.bar.baz`) over stepwise extraction across several locals (`const x = foo; const y = x.bar; ...`) when the chain is short and readable. Keep a `const` only when the value is reused, the expression is complex enough to warrant a name, or a local genuinely clarifies non-obvious logic—not to decompose a clear one-step access.
- Naming: use full words for parameters and variables—no single-letter names (`s`, `e`, `v`, `i`, …) and no opaque abbreviations (`val`, `pref`, `prefs`, `np`, `brch`, `rp`, …). Prefer domain terms (`mergeRequest`, `preference`, `value`, `username`) or descriptive names (`statusCode`, `nextPresets`); don't over-lengthen when local context already makes a shorter full word sufficient (`branch`, `project`).
- Dates: `formatDate` for relative display and `formatDateTime` for absolute strings (tooltips, etc.) in `utils.ts`; avoid ad-hoc `toLocaleString` or legacy date helpers.

## Learned Workspace Facts

- Search Merge Requests: UI in `src/components/mr_search.tsx` (entry `src/mr_search.tsx`); AI tool `src/tools/search-merge-requests.ts` is separate. `MyProjectsDropdown` with `includeAllItem={false}` (project required before load); single `List.Section` title summarizes active filters/sort (e.g. `Only open MRs (+drafts) created by me, sorted by priority`); draft toggle is `Show Drafts`/`Hide Drafts` in the Filter submenu and includes drafts without excluding other statuses; sections "Created by me", "Assigned to Review" (hidden when empty), "Other"; Sort By (`mr_sort.tsx`, ⌘S) sets GraphQL `MergeRequestSort` ("Default" omits sort). Search MR prefs use `useCachedState`; fetch uses `usePaginatedMergeRequests` → GraphQL in `mr_gql.ts`.
- `MyProjectsDropdown` in `src/components/project.tsx` supports `includeAllItem` (default `true`); My Merge Requests and Reviews keep "All Projects". My MR fetch (`useMyMergeRequests` in `mr_my.tsx`) uses `usePaginatedMergeRequests` (GraphQL `currentUser.*MergeRequests`); cache key excludes project—filter stays client-side in `MyMergeRequests`.
- My Merge Requests hides author subtitle and avatar on list rows via `showAuthor={false}` on `MRListItem`; Search MR and other MR lists keep the default (`showAuthor` true). MR list row subtitle is hidden while the side panel is open.
- MR list side panel: `useMRListDetails()` + `MRListDetailsToggleAction` (⌘⇧D). Metadata column: `useMRListMetadata()` + `MRListMetadataToggleAction` (⌘⇧I, only when side panel open). Side panel uses `MRListDetailMetadata`, full detail uses `MRDetailMetadata` in `mr_metadata.tsx`; the former `listdetails` preference was removed from `package.json`.
- Shared MR UI in `mr.tsx`, `mr_actions.tsx`, `mr_metadata.tsx`, `mr_discussions.ts`, `mr_status.ts`. MR list fetch (`mr_gql.ts` + `mr_data.ts`) includes `headPipeline`, `resolvedDiscussionsCount`, `resolvableDiscussionsCount`, and approval data in one GraphQL query—no per-row pipeline, discussion, or approval fetches on list rows. `useMRDiscussionStats` uses list counts when present; REST discussions fetch remains fallback only when counts are absent.
